Service Mesh 비교: Istio vs Cilium

항목 Istio Cilium Service Mesh
아키텍처 사이드카 프록시 (Envoy) 사이드카리스 (eBPF)
리소스 오버헤드 Pod당 Envoy 컨테이너 추가 커널 레벨, 추가 컨테이너 없음
레이턴시 ~2ms 추가 ~0.2ms 추가
L7 정책 풍부함 기본 수준
생태계 성숙도 높음 성장 중
학습 곡선 가파름 보통

Cilium Service Mesh 도입 판단

👍 장점
사이드카 제거 Pod당 Envoy 컨테이너가 없어 리소스 사용량이 크게 감소한다
레이턴시 최소화 커널 레벨에서 처리하므로 네트워크 홉이 줄어든다
CNI 통합 이미 Cilium CNI를 쓰고 있다면 추가 설치가 필요 없다
👎 단점
L7 기능 제한 Istio 대비 HTTP 라우팅, 트래픽 미러링 등 고급 기능이 부족하다
커널 의존성 Linux 커널 5.10+ 필요. 오래된 노드에서는 사용 불가
디버깅 난이도 eBPF 프로그램 문제 발생 시 Envoy보다 디버깅이 어렵다
결론
L4 중심 트래픽이 80% 이상이라면
Cilium Service Mesh가 더 적합하다
L7 정책이 핵심이라면 Istio를, 성능과 단순성이 우선이라면 Cilium을 선택한다.
두 가지를 혼합하는 것도 가능하다 (Cilium CNI + Istio ambient).
