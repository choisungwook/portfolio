Kubernetes Pod가 Pending 상태에서 빠져나오지 못하는 이유

2026. 4. 1. 14:30

왜 이 글을 쓰는가

EKS 클러스터에서 Pod를 배포했는데 Pending 상태에서 한참 멈춰있었다. kubectl describe pod를 쳐봐도 이벤트가 뭔 소린지 바로 이해가 안 됐다. 삽질 끝에 정리한 대표적인 원인 3가지를 기록한다.

원인 1: 리소스 부족

가장 흔한 원인이다. Pod의 resources.requests를 만족하는 노드가 없으면 스케줄러가 배치할 곳을 찾지 못한다.

어떻게 확인하나

Events에서 먼저 볼 문장

확인 방법은 다음과 같다.

kubectl describe pod my-pod -n default

# Events 섹션에서 이런 메시지가 보인다:
# 0/3 nodes are available: 3 Insufficient cpu.

어떻게 해결하나

가장 먼저 시도할 선택지

해결 방법은 두 가지다.

  • Pod의 resource requests를 줄인다
  • 노드를 추가한다 (Cluster Autoscaler가 있으면 자동으로 처리된다)

팁: kubectl top nodes로 현재 노드들의 리소스 사용량을 먼저 확인하면 감을 잡기 쉽다.

원인 2: nodeSelector 또는 affinity 불일치

Pod에 nodeSelector나 nodeAffinity를 설정했는데, 해당 라벨을 가진 노드가 없는 경우다. 이건 이벤트 메시지가 좀 더 구체적으로 나온다.

nodeSelector 예시

nodeSelector 예시를 보면 다음과 같다.

spec:
  nodeSelector:
    disktype: ssd

확인 명령

노드 라벨을 바로 점검하는 방법

노드에 해당 라벨이 있는지 확인하는 명령이다.

kubectl get nodes --show-labels | grep disktype

원인 3: PVC 바인딩 실패

PersistentVolumeClaim이 Bound 되지 않으면 Pod도 시작하지 못한다. StorageClass가 없거나, 가용 영역이 맞지 않는 경우가 많다.

PVC 상태 확인

먼저 Bound 여부를 본다

PVC 상태를 확인하는 명령이다.

kubectl get pvc -n default

정리

원인별 빠른 판단 기준

3가지 원인을 표로 정리하면 다음과 같다.

원인 Events 메시지 키워드 해결 방향
리소스 부족 Insufficient cpu/memory requests 조정 또는 노드 추가
nodeSelector 불일치 didn't match node selector 라벨 확인 또는 selector 수정
PVC 미바인딩 unbound PersistentVolumeClaims StorageClass, AZ 확인

대부분의 경우 kubectl describe pod의 Events 섹션을 읽으면 원인을 특정할 수 있다. 메시지가 어렵게 느껴지면 그 문장을 그대로 검색하면 된다.

코드블록 예시

Terraform

조회 예시

Terraform으로 VPC를 조회하는 HCL 코드다.

data "aws_vpc" "default" {
  default = true
}

data "aws_subnets" "default" {
  filter {
    name   = "vpc-id"
    values = [data.aws_vpc.default.id]
  }
}

Python

Pending Pod만 추리는 코드

Python으로 Pod 상태를 확인하는 스크립트다.

from kubernetes import client, config

config.load_kube_config()
v1 = client.CoreV1Api()

pods = v1.list_namespaced_pod("default")
for pod in pods.items:
    if pod.status.phase == "Pending":
        print(f"Pending: {pod.metadata.name}")

JavaScript

API 응답 필터링

JavaScript로 API 호출하는 예시다.

async function getPods(namespace = "default") {
  const response = await fetch(`/api/v1/namespaces/${namespace}/pods`);
  const data = await response.json();
  return data.items.filter(pod => pod.status.phase === "Pending");
}

Ansible

노드 라벨 적용

Ansible playbook으로 노드 라벨을 붙이는 예시다.

- name: Label Kubernetes nodes
  hosts: localhost
  tasks:
    - name: Add disktype label to worker nodes
      kubernetes.core.k8s:
        state: present
        kind: Node
        name: "{{ item }}"
        definition:
          metadata:
            labels:
              disktype: ssd
      loop:
        - worker-1
        - worker-2

Shell

CLI로 빠르게 확인

Shell 스크립트로 Pod를 확인하는 예시다.

#!/bin/bash
NAMESPACE="default"

echo "=== Pending Pods ==="
kubectl get pods -n "$NAMESPACE" --field-selector=status.phase=Pending -o wide

for pod in $(kubectl get pods -n "$NAMESPACE" --field-selector=status.phase=Pending -o name); do
  echo "--- $pod ---"
  kubectl describe "$pod" -n "$NAMESPACE" | grep -A 10 "Events:"
done

YAML로 리소스 requests를 설정하는 예시다.

apiVersion: apps/v1
kind: Deployment
metadata:
  name: nginx-deployment
  labels:
    app: nginx
spec:
  replicas: 3
  selector:
    matchLabels:
      app: nginx
  template:
    spec:
      containers:
        - name: nginx
          image: nginx:1.27
          resources:
            requests:
              cpu: "250m"
              memory: "128Mi"
            limits:
              cpu: "500m"
              memory: "256Mi"
