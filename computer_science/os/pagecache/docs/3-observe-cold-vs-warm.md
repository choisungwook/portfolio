# 같은 파일인데 두 번째 읽기만 빠른 이유 관찰

같은 1GB 파일을 두 번 읽으면 두 번째가 훨씬 빠릅니다. 그리고 읽었던 프로세스가 죽어도 그 속도는 유지됩니다. page cache가 프로세스가 아니라 커널의 소유라는 것을 눈으로 확인합니다.

환경 준비는 [1. 실습 준비](./1-setup.md)를 따릅니다.

## cold read: 캐시가 비어 있을 때

캐시를 비운 직후의 Cached 크기를 확인하고, 파일을 읽는 시간을 잽니다.

```bash
sync && echo 3 > /proc/sys/vm/drop_caches
grep "^Cached" /proc/meminfo
time cat /tmp/pagecache-test > /dev/null
grep "^Cached" /proc/meminfo
```

실행 결과 예시입니다. 읽기 전 Cached는 약 223MB였는데, 1GB 파일을 읽고 나니 약 1,247MB로 정확히 파일 크기만큼 늘었습니다. 파일 데이터가 page cache에 들어갔다는 뜻입니다.

```text
Cached:           228916 kB

real    0m0.343s

Cached:          1277688 kB
```

## warm read: 캐시에 있을 때

캐시를 비우지 않고 같은 파일을 다시 읽습니다.

```bash
time cat /tmp/pagecache-test > /dev/null
```

같은 명령이 0.343초에서 0.139초로 빨라졌습니다. 이번에는 디스크에 가지 않고 메모리 사본만 복사했기 때문입니다.

```text
real    0m0.139s
```

이 환경은 빠른 NVMe라 2~3배 차이지만, HDD나 클라우드 네트워크 볼륨(EBS gp3 등)에서는 수십 배 이상 벌어집니다.

## 캐시는 프로세스가 죽어도 남는다

위 실험에는 이미 중요한 사실이 숨어 있습니다. 첫 번째 cat과 두 번째 cat은 **서로 다른 프로세스**입니다. 첫 cat은 파일을 읽고 이미 종료됐는데, 두 번째 cat이 그 캐시 덕을 봤습니다. 캐시가 프로세스 안(heap 등)에 있었다면 프로세스 종료와 함께 사라졌을 것입니다.

여기서 보통 "그럼 저 1GB는 누가 점유하고 있는 건가"라고 묻습니다. 커널입니다. free로 보면 어떤 프로세스의 사용량도 아닌 cache 열에 잡혀 있고, 다른 프로세스가 메모리를 필요로 하면 커널이 캐시를 줄여서 내어 줍니다.

```bash
free -wh
```

Kafka가 자체 캐시 대신 page cache를 택한 결정적 이유가 이것입니다. broker 프로세스를 재시작해도 캐시는 커널에 그대로 남아 있어서, 재시작 직후에도 warm read 속도로 서비스할 수 있습니다. heap 안에 캐시를 만들었다면 재시작할 때마다 캐시를 처음부터 다시 채워야 합니다.

## 정리

파일을 읽으면 커널이 그 데이터를 page cache에 남기고, 다음 읽기는 프로세스가 누구든 메모리 속도로 처리됩니다. 캐시의 수명이 프로세스가 아니라 커널(머신)에 묶여 있다는 것이 애플리케이션이 자체 캐시 없이 OS에 기대도 되는 근거입니다.
