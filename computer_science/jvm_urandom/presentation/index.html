1. 왜 이 옵션을 썼나
증상: headless VM에서 Tomcat 기동이 수 분 멈춘다
SHA1PRNG의 최초 seed 읽기가 /dev/random에서 blocking
// catalina.out Creation of SecureRandom instance for session ID generation using [SHA1PRNG] took [193,502] ms.
①
JVM (Tomcat)
SHA1PRNG
② seed 요청
/dev/random
(blocking pool)
③
entropy 고갈 → read가 멈춘다
✕
keyboard·mouse가 없는 VM은
entropy가 쌓이지 않는다
해결: -Djava.security.egd=file:/dev/./urandom
→ seed 소스를 blocking하지 않는 urandom으로 변경

2. 점 하나의 정체
정답: JDK 7 이하의 문자열 special-case를 우회하는 트릭
JDK-4705093: "file:/dev/urandom" 문자열이면 설정을 무시하고 /dev/random을 읽었다 (JDK-6202721)
egd=file:/dev/urandom
문자열 일치 → 무시
/dev/random
✕
여전히 blocking
egd=file:/dev/./urandom
비교식 빗나감
/dev/urandom
같은 파일, 다른 문자열 → 즉시 seed 확보
※ JDK 8에서 special-case가 수정되어 점 없는 경로도 동작한다

최신 커널에서는
3. 최신 커널에서는 — 옵션이 필요 없다
Linux 5.6(2020.3)에서 blocking pool이 제거됐다
CRNG 초기화 이후 /dev/random과 /dev/urandom은 같은 CRNG 출력 (5.18에서 내부 통합)
kernel < 5.6
/dev/random
entropy 추정치로 blocking
/dev/urandom
blocking 없음
➡
kernel ≥ 5.6 (AL2023 = 6.1)
/dev/random ＝ /dev/urandom
CRNG 초기화 후 둘 다 blocking 없음
이 옵션은 두 겹으로 화석 — JDK 8 이상이면 점(.)이 불필요, kernel 5.6 이상이면 옵션 전체가 불필요

마치며
5. 마치며
요약
옵션은 구버전 커널 /dev/random blocking 회피용, 점(.)은 JDK 7 이하 special-case 우회용
kernel 5.6 이상 + JDK 8 이상이면 둘 다 불필요 — AL2023(6.1)은 해당 없음
macOS·Docker는 커널을 못 바꿔서 재현 불가, kernel 4.14 EC2에서 재현 가능
다음 공부
getrandom(2) syscall과 CRNG 초기화 과정, container의 /dev 장치 mount 원리
레거시 옵션을 지우기 전에 uname -r부터 확인하자. 감사합니다!
