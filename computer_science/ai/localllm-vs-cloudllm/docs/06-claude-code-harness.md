# 로컬 모델로 Claude Code harness 활용하기

저는 [05-lmstudio-claude-code.md](./05-lmstudio-claude-code.md) 에서 LM Studio를 Claude Code에 붙이는 데까지 성공했습니다. 그런데 막상 붙이고 나니 "Claude Code의 슬래시 커맨드, skill, agent 같은 기능을 로컬 모델로도 쓸 수 있나?"라는 질문이 따라왔습니다. 이 글은 그 질문에 답을 찾으면서 정리한 메모입니다. 2026년 4월 기준입니다.

## Claude Code harness가 뭘까

먼저 용어 정리부터 하겠습니다. 제가 이 글에서 쓰는 **harness**는 Claude Code가 모델 위에 얹어둔 실행 환경 전체를 뜻합니다. 구체적으로는 다음을 포함합니다.

- **tool**: 파일 읽기/쓰기, bash 실행, 웹 검색 같은 외부 동작
- **slash command**: `/clear`, `/init` 같은 명령어
- **skill**: 특정 상황에서 모델이 따라야 하는 작업 묶음
- **agent**: 별도 컨텍스트에서 작업을 위임받는 sub-agent
- **hook**: 특정 이벤트(예: 도구 호출 직후)에 끼어드는 스크립트

이 전부가 "모델이 도구를 호출하고, 그 결과를 받아 다음 행동을 결정하는" 흐름을 전제로 동작합니다. 그래서 모델의 **tool use 능력**이 harness 활용도를 결정합니다.

## 그래서 작은 로컬 모델로는 어디까지 되나

저는 Gemma 4 E2B로 일부 항목을 시도했고, 시도하지 못한 항목은 일반적인 작은 모델의 tool use 한계에서 추정한 결과입니다. 표에서 "직접 시도"와 "추정"을 구분했습니다.

| harness 기능 | 작은 로컬 모델(2B 이하) | 검증 출처 |
|---|---|---|
| 일반 채팅 | 가능 | 직접 시도 |
| Read tool로 파일 읽기 | 거의 안 됨 | 직접 시도 |
| Edit/Write tool로 파일 수정 | 거의 안 됨 | 추정 (tool 호출 자체가 안 되니 자연 따라옴) |
| Bash 명령 실행 | 거의 안 됨 | 추정 |
| TodoWrite로 작업 추적 | 거의 안 됨 | 추정 |
| 슬래시 커맨드(`/clear` 등) | 가능 | 직접 시도, harness가 처리하므로 모델 무관 |
| skill 자동 호출 | 불안정 | 추정 (skill도 tool 호출 기반) |
| sub-agent 호출 | 거의 안 됨 | 추정 |

요약하면 **slash command와 일반 채팅 정도가 안정적**이고, 그 외에는 모델이 따라가지 못합니다. 저는 이걸 처음 봤을 때 살짝 실망했지만, 작은 모델의 한계를 정직하게 인정하는 게 맞다고 정리했습니다.

## 그럼에도 작은 모델로 살릴 수 있는 사용 케이스

쓸모가 없냐 하면 그렇진 않습니다. 저는 다음과 같은 작업에 작은 로컬 모델 + Claude Code 조합을 씁니다.

### 회사 코드 한 조각의 의도 파악

저는 회사 코드를 직접 붙여넣고 짧게 묻는 식으로 씁니다. 회사 코드 자체는 본문에 실을 수 없어서 다음과 같은 형태로만 적습니다.

```text
이 함수가 무슨 일을 하는지 한 문장으로 설명해줘.
<여기에 사내 코드 paste>
```

응답은 보통 2~3문장 정도이고, tool use 없이 텍스트 in / 텍스트 out만으로 끝납니다. 외부 API에 보낼 수 없는 코드를 빠르게 이해할 때 유용합니다.

### 사내 문서 요약

회사 위키나 RFC 문서 일부를 붙여넣고 요약을 받습니다. 외부에 못 보내는 문서일수록 이 조합의 가치가 큽니다.

### 영문 메시지 번역과 톤 다듬기

짧은 영문 응답을 자연스러운 한국어로 바꾸거나 그 반대로 바꾸는 작업도 잘 됩니다.

이 세 가지의 공통점은 **tool 호출이 필요 없다**는 점입니다. 모델이 텍스트 in / 텍스트 out만 잘하면 되는 작업입니다.

## harness를 제대로 쓰려면 어떤 모델이 필요할까

LM Studio 공식 문서가 권장하는 기준은 두 가지입니다.

- **context length 25k 토큰 이상**: harness는 tool 호출 결과까지 같이 컨텍스트에 넣기 때문에 컨텍스트가 빨리 소진됩니다.
- **tool use(function calling) 안정 지원**: 모델이 JSON 형식의 도구 호출을 안정적으로 만들어낼 수 있어야 합니다.

이 둘을 만족하는 모델은 보통 **20B급 이상**입니다. LM Studio 공식 문서가 예시로 드는 `gpt-oss-20b`도 이 라인입니다. 제 노트북에서는 이 크기 모델이 무리라서 저는 다음 두 가지 옵션을 후보로 두고 있습니다.

- **사내 GPU 서버에 큰 모델을 띄우고 원격으로 연결** — LM Studio 공식 문서가 LM Link라는 이름의 원격 실행 기능을 언급합니다. 저는 사내 GPU가 없어서 이 기능을 직접 써본 적은 없습니다.
- **클라우드의 GPU 인스턴스에 LM Studio를 띄우고 SSH 터널로 연결** — 데이터가 외부 클라우드로 가긴 하지만, 회사가 운영하는 자체 클라우드(VPC 내) 안에서라면 정책상 허용될 수 있습니다. 저희 회사 정책으로는 가능한 범위인지 아직 확인하지 못했고, 일반론으로 제시하는 옵션입니다.

## 추천 워크플로우

저는 지금 다음과 같이 나눠 쓰고 있습니다. 절대적인 기준은 아니고, 각자의 정책과 하드웨어에 맞게 조정하시면 됩니다.

| 상황 | 도구 | 모델 | 비고 |
|---|---|---|---|
| 회사 코드를 단순 질의응답 (tool 호출 불필요) | LM Studio + Claude Code | Gemma 4 E2B 같은 작은 모델 | 직접 운영 중 |
| 회사 코드를 파일 단위로 수정 | LM Studio + Claude Code | 사내 GPU에 띄운 20B 이상 모델 | 미실측, 향후 계획 |
| 공개 코드, 사이드 프로젝트 | Claude Code (cloudllm) | Anthropic 공식 모델 | 직접 운영 중 |
| 오프라인 환경 (비행기 등) | LM Studio + Claude Code | 작은 모델로 텍스트 질의응답만 | 직접 운영 중 |

## 그래서 결론

저의 결론은 단순합니다. **작은 로컬 모델 + Claude Code 조합은 "축소된 채팅 도구"입니다.** harness의 진가는 tool 호출에서 나오는데, 그건 모델이 클수록 잘 됩니다. 작은 모델로는 harness의 전부를 활용할 수 없다는 사실을 인정하고, 텍스트 질의응답에 한정해서 쓰는 게 정직한 사용법이라고 봅니다.

그렇다고 의미가 없는 건 아닙니다. 회사 코드를 외부에 보낼 수 없는 상황에서 "축소된 채팅 도구"라도 손에 있는 것과 없는 것의 차이는 큽니다. 저는 이 한계를 인정한 채로 매일 쓰고 있고, 더 큰 모델이 손에 들어오면 그때 비율을 바꿀 생각입니다.

## 더 공부할 것

- `gpt-oss-20b`나 비슷한 20B급 모델을 사내 GPU에 띄워서 tool use가 실제로 어디까지 안정적인지 측정해보고 싶습니다.
- Claude Code의 sub-agent를 작은 로컬 모델 + 큰 클라우드 모델로 분기시키는 hybrid 패턴이 가능한지도 시도해보지 못했습니다.
