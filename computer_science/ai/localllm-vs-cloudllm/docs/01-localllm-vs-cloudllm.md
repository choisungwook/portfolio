# localllm과 cloudllm의 차이

저는 Claude Code를 매일 쓰면서 회사 일과 사이드 프로젝트를 같이 돌립니다. 그런데 2026년 초에 보안 담당자님께서 "회사 코드는 외부 LLM API로 보내지 마세요"라는 공지를 주셨고, 그때부터 저는 cloudllm만 쓰던 환경에서 localllm을 다시 들여다보게 되었습니다. 이 글은 그 경험을 정리한 글입니다.

## 두 단어의 정의

먼저 용어부터 짚겠습니다. **cloudllm**은 외부 회사가 운영하는 거대 모델을 API로 호출해서 쓰는 방식입니다. Claude API, OpenAI API, Gemini API가 여기에 해당합니다. **localllm**은 제 노트북이나 사내 GPU 서버에서 모델을 직접 띄워서 쓰는 방식입니다. 모델 파일을 디스크에 받아두고 로컬 프로세스로 추론을 돌립니다. 두 방식의 차이를 결정짓는 건 결국 모델이 어디에서 돌아가는가 하나로 모입니다.

## 비용

저는 2025년에 Claude API를 거의 매일 호출했고, 어떤 달은 토큰 비용이 제 통신비보다 더 나오기도 했습니다. cloudllm의 비용 구조는 토큰 단위 과금이라서 쓰면 쓸수록 누적됩니다. 반면 localllm은 처음에 하드웨어 비용을 내고 나면 그 뒤로는 추가 비용이 거의 없습니다. 제 경우엔 이미 가지고 있던 M 시리즈 Mac에서 돌렸기 때문에 추가 비용은 0원이었습니다.

다만 이 비교는 단순하지 않습니다. localllm을 진지하게 쓰려면 결국 더 큰 모델을 돌릴 GPU가 필요하고, 회사 환경에서는 사내 GPU 서버 임대 비용이 또 들어갑니다. 그래서 저는 비용을 단순히 "무료냐 유료냐"로 보지 않고, "내가 쓸 수 있는 모델의 크기에 비례한 비용"이라고 이해했습니다.

## 데이터 보안

이게 제가 localllm을 다시 보게 된 가장 큰 이유였습니다. cloudllm을 쓰면 제가 입력한 코드와 프롬프트가 전부 외부 API로 나갑니다. 회사 정책상 그게 금지된 순간 저는 cloudllm을 회사 코드에 쓸 수 없게 되었습니다. localllm은 모든 추론이 제 노트북 안에서 끝나기 때문에 외부로 데이터가 나가지 않습니다. 보안 담당자님께서 안심하시는 부분이 정확히 이 지점이었습니다.

## 인터넷 연결

저는 가끔 카페에서 코드를 만지는데, 와이파이가 불안정한 곳에서는 cloudllm 응답이 끊깁니다. localllm은 모델 파일이 이미 디스크에 있기 때문에 인터넷이 없어도 동작합니다. 비행기 안이나 지하철에서 작업할 때 이게 생각보다 큰 차이로 다가왔습니다.

## 성능과 모델 크기

이 부분에서 저는 localllm의 한계를 가장 크게 느꼈습니다. 2026년 4월 기준 cloudllm은 수백 B(billion) 단위 파라미터를 가진 거대 모델을 제공합니다. 반면 제 맥북에서 무리 없이 돌릴 수 있는 localllm은 보통 2B~12B 정도이고, 더 큰 모델을 돌리려면 노트북이 뜨거워지거나 응답이 느려집니다. 한 번은 Gemma 3 27B 모델을 받아 돌렸다가 응답 한 줄이 나오기까지 거의 1분이 걸려서 결국 지웠습니다. 모델 크기 차이가 곧 답변 품질 차이로 이어졌고, 복잡한 리팩터링 같은 작업은 cloudllm이 확실히 더 잘했습니다.

## 응답 속도

cloudllm의 응답 시간은 네트워크 latency와 외부 서버 부하의 영향을 받습니다. localllm은 그게 없는 대신 제 하드웨어 성능에 직접 묶입니다. 제 경험으로는 작은 모델(2B~4B)을 M 시리즈 Mac에서 돌리면 첫 토큰이 나오기까지의 지연이 cloudllm보다 짧았습니다. 다만 모델이 커질수록 이 장점은 빠르게 사라졌습니다.

## 운영 부담

cloudllm은 운영 부담이 거의 없습니다. API 키 하나 받아두면 끝이고, 모델 업데이트도 외부 회사가 알아서 합니다. localllm은 그 반대입니다. 제가 직접 모델을 받고, 디스크 공간을 관리하고, 새 버전이 나오면 다시 받아야 합니다. 모델 파일 하나가 보통 1GB~수십 GB라 디스크가 금방 찹니다. 저는 LM Studio를 쓰면서 모델 폴더 위치를 외장 SSD로 옮겨두는 식으로 관리하고 있습니다.

## 한눈에 비교

지금까지 정리한 차이를 표로 모았습니다. 한 번에 보고 싶을 때 이 표만 보셔도 됩니다.

| 항목 | localllm | cloudllm |
|---|---|---|
| 비용 구조 | 초기 하드웨어 비용 (그 뒤 거의 0) | 토큰 단위 과금 (사용량 누적) |
| 데이터 | 외부로 나가지 않음 | 외부 API로 전송 |
| 인터넷 | 오프라인 가능 | 항상 필요 |
| 모델 크기 | 보통 2B~12B (소비자 하드웨어 기준) | 수백 B 단위 |
| 응답 속도 | 하드웨어 성능에 좌우 | 네트워크 latency 포함 |
| 운영 부담 | 모델 다운로드, 디스크 관리 | API 키만 관리 |
| 답변 품질 | 자동완성·짧은 리팩터링은 충분, 복잡한 리팩터링은 부족 | 대체로 안정적으로 좋음 |

## 그래서 언제 어떤 걸 쓰나

저는 정답이 없다고 생각합니다. 같은 질문이라도 회사 정책, 작업의 민감도, 지금 가진 하드웨어, 비용 한도에 따라 답이 달라지기 때문입니다. 지금 제가 쓰는 기준은 다음과 같습니다.

회사 코드를 다룰 때는 localllm을 씁니다. 보안 정책상 그래야 하고, 작업이 자동완성이나 짧은 리팩터링 위주라 작은 모델로도 충분합니다. 사이드 프로젝트나 공개 코드를 다룰 때, 또는 복잡한 설계 회의를 할 때는 cloudllm을 씁니다. 모델이 똑똑할수록 시간이 절약되고, 외부로 나가도 문제없는 데이터이기 때문입니다.

이 기준이 모두에게 맞는 답은 아닙니다. 누군가는 사내 GPU 서버에서 70B급 모델을 돌리고 있을 수 있고, 누군가는 cloudllm 토큰 비용이 회사 차원에서 보장되어 있을 수 있습니다. 제 글은 "저는 이런 기준으로 선택했다"의 한 사례로 읽어주시면 좋겠습니다.

## 더 공부할 것

- 사내 GPU 서버에 큰 모델(예: 30B 이상)을 띄웠을 때 cloudllm과의 품질 격차가 얼마나 좁혀지는지 직접 비교해보고 싶습니다.
- localllm에서 같은 모델을 quantization 단계만 바꿔가며 (Q4, Q8, FP16) 답변 품질을 비교하는 실험도 아직 못 해봤습니다.
