# JVM Warm-up 이론

이 실습의 질문은 두 가지다.

- warm-up이 Hikari `acquire time`을 줄였는가
- 첫 요청의 병목이 애플리케이션 부팅 직후 초기화 비용 때문인가

핵심은 `acquire time`을 먼저 보는 것이 아니라 `usage time`을 먼저 보는 것이다.

## 왜 첫 요청에서 준비 비용이 붙나

Java 애플리케이션이 시작될 때 모든 `.class` 파일이 한꺼번에 실행 준비를 끝내는 것은 아니다. JVM은 보통 "지금 실제로 필요한 것부터" 준비한다. 이 성질을 이해해야 첫 요청이 왜 느려질 수 있는지 설명할 수 있다.

가장 바깥 개념은 JVM의 lazy class loading이다. 클래스는 프로그램 시작 순간 전부 메모리에 올라오는 것이 아니라, 해당 코드 경로가 실제로 필요해질 때 로딩된다. 이때 JVM은 보통 아래 단계를 거친다.

- `Loading`: `.class` 파일을 읽어 메모리에 올린다
- `Linking`: 클래스 구조를 검증하고 참조를 연결한다
- `Initialization`: `static` 초기화와 정적 필드 준비를 수행한다

즉, 애플리케이션 프로세스가 이미 떠 있어도 어떤 컨트롤러, 서비스, 직렬화 경로, 쿼리 경로는 아직 "처음 실행되지 않은 상태"일 수 있다.

Spring Boot도 비슷하다. 애플리케이션 시작 시점에 빈을 등록하고 웹 서버를 열지만, 모든 요청 조합을 실제로 한 번씩 실행해서 끝까지 예열하지는 않는다. 그래서 아래는 서로 다른 이야기다.

- 애플리케이션이 기동됐다
- 특정 API 경로가 완전히 warm 상태다

이 실습이 보는 문제는 두 번째다.

## 부팅 직후 첫 요청에서 무슨 일이 생기나

Spring Boot 앱이 `UP` 상태가 되더라도 모든 코드 경로가 완전히 데워진 것은 아니다. JVM과 프레임워크는 필요한 시점까지 일부 준비를 미룬다. 그래서 첫 사용자 요청이 들어오면 그 요청 안에서 아래 작업이 한 번에 겹칠 수 있다.

여기서 말하는 초기화 비용은 단순히 "코드가 느리다"는 뜻이 아니라, 첫 실행 시점에만 추가로 드는 준비 비용이다.

- JVM 클래스 로딩, 링크, 초기화
- Spring MVC 핸들러/검증 경로의 첫 실행 준비
- Hibernate JPQL 파싱과 쿼리 플랜 생성
- JPA 엔티티/DTO 매핑 경로 준비
- Jackson 직렬화 메타데이터 준비

JVM은 클래스를 프로그램 시작 시 전부 올리지 않는다. 실제로 해당 코드 경로를 처음 지날 때 필요한 클래스와 메타데이터를 준비한다. Spring Boot도 애플리케이션을 띄울 때 모든 요청 경로를 끝까지 한 번씩 실행하지는 않는다. 그래서 애플리케이션은 이미 떠 있지만, 특정 API 입장에서는 아직 cold 상태일 수 있다.

정리하면 첫 요청 시간은 아래처럼 나뉜다.

```text
첫 요청 처리 시간 = 초기화 비용 + 실제 비즈니스 처리 시간
이후 요청 처리 시간 = 실제 비즈니스 처리 시간
```

이 초기화 비용은 보통 첫 몇 요청에서만 나타난다. 이후 같은 요청은 이미 준비된 경로를 재사용하므로 더 짧아진다.

## 왜 Hikari `usage`가 먼저 늘어나나

이 앱의 주 실험 경로는 읽기 요청 하나가 다음 단계를 지난다.

1. Controller에서 파라미터 검증
2. Service의 read-only 트랜잭션 시작
3. 여러 JPA 조회 실행
4. DTO 조립
5. 응답 직렬화

첫 요청에서 Hibernate와 매핑 경로 준비 비용이 트랜잭션 안쪽에서 일어나면, 커넥션을 빌린 뒤 반환하기까지의 시간이 길어진다. 이 값이 Hikari `usage time`이다.

즉, 해석 순서는 아래처럼 잡는 것이 맞다.

```text
부팅 직후 초기화 비용 증가
-> 같은 요청의 connection usage time 증가
-> 동시에 처리 가능한 커넥션 수 감소
-> pending / acquire time 증가
```

그래서 `acquire time`이 커졌다고 해서 곧바로 DB가 느리다고 보면 안 된다. 먼저 `usage time`이 왜 늘었는지 봐야 한다.

## warm-up이 하는 일

이 실습의 warm-up은 앱이 준비된 직후 `ApplicationReadyEvent`에서 자기 자신에게 실제 검색 API를 HTTP로 호출한다. 중요한 점은 “실제 사용자 요청과 같은 경로”를 미리 실행한다는 것이다.

- DispatcherServlet
- Controller validation
- Service transaction
- Repository/JPA query path
- DTO mapping
- Jackson serialization

이 경로를 사용자 요청 전에 먼저 통과시키면, 첫 burst가 들어왔을 때 초기화 비용이 이미 대부분 소진된 상태가 된다. 그 결과 `usage time`이 먼저 안정되고, 뒤따르던 `acquire time` 상승도 줄어든다.

## 이 실습에서 기대하는 관찰

- `no-warmup`
  - 부팅 직후 burst에서 API latency max와 p95가 먼저 튄다
  - 같은 구간에서 Hikari `usage time`이 증가한다
  - 이어서 `pending`, `acquire time`이 뒤따라 올라간다
- `with-warmup`
  - 같은 burst에서도 초기 spike가 더 작다
  - 특히 `usage time`이 먼저 안정되므로 `acquire time`도 덜 오른다

이 패턴이 보이면, 첫 요청 병목을 Hikari 설정 하나의 문제로 보기보다 “애플리케이션 부팅 직후 초기화 비용이 커넥션 사용 시간을 밀어 올린 결과”로 해석할 수 있다.
