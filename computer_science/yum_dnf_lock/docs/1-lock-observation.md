# yum/dnf lock은 왜 동시에 하나만 허용할까?

패키지 설치는 단순히 파일 하나를 내려받는 작업처럼 보이지만, 실제로는 RPM 데이터베이스와 repo metadata를 함께 바꿉니다. 그런데 두 프로세스가 동시에 이 상태를 바꾸면 어떤 문제가 생길까요?

## 왜 lock이 필요한가?

dnf는 패키지 설치, 삭제, metadata 갱신 과정에서 같은 시스템 상태를 읽고 씁니다. 이때 한 프로세스가 transaction을 계산하는 동안 다른 프로세스가 RPM database나 cache를 바꾸면 첫 번째 프로세스가 계산한 결과가 더 이상 맞지 않을 수 있습니다.

그래서 dnf는 작업 구간에 lock을 잡습니다. **lock은 패키지 매니저가 느려서 생기는 부작용이 아니라, 같은 시스템 상태를 한 번에 한 프로세스만 바꾸게 하는 안전장치입니다.**

## lock이 걸리면 무엇을 관찰해야 할까?

관찰 대상은 세 가지입니다.

- 어떤 dnf 프로세스가 먼저 실행 중인지 확인합니다.
- `/var/cache/dnf`, `/var/lib/dnf` 아래 lock 또는 pid 후보 파일을 확인합니다.
- 두 번째 dnf 명령이 바로 실패하는지, 아니면 기존 프로세스가 끝나기를 기다리는지 확인합니다.

이 저장소의 실습은 느리게 응답하는 local repo를 일부러 만들고, 첫 번째 dnf가 metadata를 받는 동안 두 번째 dnf를 실행합니다. 외부 패키지 저장소 상태에 의존하지 않고 lock 경합만 보기 위한 구성입니다.

## 장단점은 무엇인가?

Docker Compose로 재현하는 장점은 로컬 머신의 실제 yum/dnf 상태를 건드리지 않는다는 점입니다. 반복 실행과 정리도 쉽습니다.

단점은 컨테이너 이미지의 dnf 버전에 따라 lock 메시지와 파일 경로가 조금 달라질 수 있다는 점입니다. 그래서 문서에서는 특정 파일명 하나를 정답으로 두지 않고, 프로세스와 lock 후보를 같이 확인합니다.
