# yum/dnf lock은 왜 필요할까

패키지 설치는 파일 복사보다 위험한 작업입니다.

`dnf install`은 실행 파일만 놓고 끝나지 않습니다. rpm database, metadata cache, transaction history처럼 패키지 상태를 설명하는 데이터도 함께 바꿉니다. 이 글은 yum/dnf lock이 왜 필요한지, lock 대기를 왜 장애로만 보면 안 되는지 짧게 정리합니다.

실습 준비는 [1. 실습 준비](./1-setup.md)를 따릅니다.

## 패키지 매니저는 무엇을 보호해야 할까요?

**yum/dnf lock은 패키지 상태를 한 번에 하나의 transaction만 바꾸게 만드는 안전장치입니다.**

한 프로세스가 패키지 A를 설치하면서 rpm database를 갱신하는 중이라고 가정합니다. 이때 다른 프로세스가 동시에 패키지 B를 설치하면, 두 프로세스가 서로의 중간 상태를 볼 수 있습니다. 설치 완료 여부, 의존성 해석, history 기록이 어긋날 수 있습니다.

## lock 대기는 왜 정상일까요?

lock 대기는 느려 보이지만, 패키지 상태를 깨뜨리지 않기 위한 정상 동작입니다. lock을 사용하면 설치, 삭제, metadata 갱신 작업이 서로의 중간 상태를 밟지 않습니다.

단점도 있습니다. 자동화 스크립트가 이미 실행 중인 패키지 작업을 모르고 또 실행하면 배포가 멈춘 것처럼 보입니다. 그래서 운영에서는 lock 대기를 바로 장애로 단정하지 않고, 먼저 누가 lock을 잡고 있는지 확인해야 합니다.

## yum과 dnf의 lock 경로는 항상 같을까요?

항상 같다고 단정하면 안 됩니다. 배포판, dnf/yum 버전, 작업 종류에 따라 metadata lock, rpmdb lock, process pid file 위치가 달라질 수 있습니다.

이 핸즈온은 Amazon Linux 2023 기준으로 `/var/cache/dnf/metadata_lock.pid`를 기본 후보로 사용합니다. 다른 환경에서 결과가 다르면 lock 경로를 먼저 확인해야 합니다. 배포판별 차이는 `확인 필요`입니다.

## 무엇을 봐야 할까요?

yum/dnf lock은 설치 명령을 방해하는 파일이 아니라 패키지 상태를 보호하는 직렬화 장치입니다. lock 경합이 보이면 어떤 프로세스가 어떤 lock 파일을 잡고 있는지 확인하는 쪽으로 넘어가야 합니다.
