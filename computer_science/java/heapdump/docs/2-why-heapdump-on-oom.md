# 2. OOM은 재현되지 않는다 — JVM이 죽는 순간을 남기는 옵션

OOM 알림을 받고 들어가 보면 프로세스는 이미 재시작됐고 heap은 비어 있습니다. 원인을 알려줄 증거가 사라진 뒤입니다. 그런데 JVM에는 죽는 순간의 heap 전체를 파일로 남기는 옵션이 이미 들어 있고, 평소 실행 비용은 없습니다. dump 한 장이 왜 원인 분석의 전부를 담는지부터, 왜 이 옵션이 기본 세팅처럼 쓰이는지까지 정리합니다.

## 왜 객체는 전부 heap에 살까요?

heap을 치우는 주체는 개발자가 아닙니다. heap은 유한한데 객체는 계속 생성되므로, 안 쓰는 객체를 회수하지 않으면 금방 고갈됩니다. 그래서 JVM의 garbage collection(GC)이 더는 쓰이지 않는 객체를 찾아 회수합니다. "쓰이지 않는다"의 판별 기준은 참조 도달성입니다. GC는 GC root(실행 중인 스레드의 지역 변수, static 필드 등)를 출발점으로 참조 사슬을 따라가, 닿는 객체만 남기고 닿지 않는 객체를 회수합니다.

이 구조가 OOM의 정확한 의미를 정해 줍니다. GC는 평소에는 heap 전체가 아니라 새 객체가 모이는 일부 영역(young generation)만 골라 회수합니다. GC가 도는 동안 애플리케이션이 잠깐 멈추는데, heap 전체를 훑으면 이 멈춤이 길어지기 때문입니다. 자리가 없으면 마지막 수단으로 heap 전체를 훑는 full GC를 돌립니다. 그래도 자리를 못 만들면 그제서야 OOM을 던집니다.

객체 생성부터 OOM까지의 흐름은 이렇습니다.

객체 생성 요청
→ 자리 없으면 young generation만 정리 (minor GC)
→ 그래도 자리 없으면 heap 전체 정리 (full GC)
→ 그래도 자리 없으면 OOM (java.lang.OutOfMemoryError: Java heap space)

**heap OOM은 "아직 살아 있는(참조에 닿는) 객체만으로 heap이 가득 찬 상태"입니다.** 원인은 둘 중 하나입니다 — 놓아줬어야 할 참조가 객체를 붙잡고 있거나(leak), 정말 필요한 데이터가 heap 크기를 넘거나(heap 부족).

예를 들어 static으로 선언한 Map이나 List가 그렇습니다. static 변수는 GC root, 즉 GC가 "살아 있는 객체"를 찾기 시작하는 출발점 중 하나라서 프로그램이 끝날 때까지 계속 남아 있습니다. 그래서 여기에 데이터를 넣기만 하고 지우지 않으면, GC root에서 참조가 닿는 상태이므로 아무도 안 쓰더라도 GC가 영영 치우지 못합니다.

## 왜 그 답이 heap dump에 있을까요?

heap dump는 그 순간 heap의 모든 객체와 객체 사이의 참조 관계를 통째로 담은 스냅샷 파일(HPROF 바이너리 형식, 확장자 `.hprof`)입니다. 객체가 전부 heap에 살기 때문에 heap 스냅샷 하나가 곧 애플리케이션이 들고 있던 데이터 전부이고, 참조 그래프가 있기 때문에 쌓인 객체에서 사슬을 거꾸로 따라가 붙잡고 있는 지점(예: static 컬렉션)까지 역추적할 수 있습니다. OOM 직전의 full GC가 죽은 객체를 이미 정리했으므로, dump에 남은 것은 대부분 살아 있는 용의자 목록이기도 합니다.

메트릭과 GC 로그로 부족한 이유가 여기서 나옵니다. 메트릭은 "얼마나"(총량 추이), GC 로그는 "언제"(회수 시점과 빈도)를 알려주지만, 참조 그래프는 어느 쪽에도 없습니다. "무엇이, 누구 때문에" 남아 있는지는 heap dump만 답할 수 있습니다.

## 왜 "OOM 순간"의 dump여야 할까요?

메모리 누수로 인한 OOM은 대부분 재현이 어렵습니다. 특정 트래픽 패턴, 특정 데이터, 며칠의 누적이 겹쳐야 터지기 때문입니다. 그래서 "터지면 그때 dump를 뜨자"는 계획은 성립하지 않습니다. 터지는 시점을 사람이 모르고, 터진 뒤에는 프로세스 재시작과 함께 증거가 사라집니다.

`-XX:+HeapDumpOnOutOfMemoryError`는 이 문제를 뒤집습니다. JVM이 OutOfMemoryError를 던지는 바로 그 시점에 스스로 dump를 남깁니다. 사람이 없는 새벽에 터져도 증거가 남습니다.

여기서 보통 "옵션을 켜두면 평소 성능에 영향이 있지 않나"를 묻습니다. 없습니다. 이 옵션은 OOM이 발생하는 코드 경로에서만 동작하는 수동적인 플래그라서, OOM이 없는 평상시에는 아무 일도 하지 않습니다. 비용은 OOM이 터져 dump 파일을 쓰는 그 순간에만 발생합니다.

## 함께 알아야 할 JVM 옵션

실습과 운영에서 함께 쓰는 옵션은 다음과 같습니다.

| 옵션 | 동작 |
| --- | --- |
| `-XX:+HeapDumpOnOutOfMemoryError` | 첫 OOM 발생 시 heap dump를 쓴다. 같은 JVM 프로세스에서 두 번째 OOM부터는 쓰지 않는다 |
| `-XX:HeapDumpPath=<경로>` | dump 저장 위치. 디렉터리를 주면 `java_pid<pid>.hprof`로, 파일 경로를 주면 그 이름으로 쓴다. 기본값은 JVM의 작업 디렉터리 |
| `-XX:+ExitOnOutOfMemoryError` | 첫 OOM에서 JVM을 즉시 종료한다. dump 옵션과 같이 쓰면 dump를 먼저 쓰고 종료한다 |
| `-XX:+CrashOnOutOfMemoryError` | 첫 OOM에서 JVM을 crash시켜 core dump와 hs_err 파일을 남긴다 |
| `-XX:OnOutOfMemoryError=<명령>` | 첫 OOM에서 지정한 명령을 실행한다. 알림 연동 정도에 쓴다 |

"첫 OOM에서 한 번만"은 이 옵션의 성격을 잘 보여줍니다. 목적이 지속적인 관측이 아니라 사고 순간의 증거 확보이기 때문입니다. 실행 중인 JVM을 관찰하고 싶을 때는 `jcmd <pid> GC.heap_dump`로 원하는 시점에 수동 dump를 뜹니다. 자동 dump와 수동 dump는 대체재가 아니라 보완재입니다.

이 "한 번"의 단위는 JVM 프로세스입니다. 한 번 썼다는 사실을 JVM 내부 플래그로 기억하는데, 이 플래그는 프로세스 메모리에만 있습니다. 그래서 systemd나 orchestrator가 죽은 프로세스를 재시작하면 새 JVM에서 플래그가 초기화된 상태로 시작하고, 새 프로세스가 다시 OOM을 만나면 다시 1회 dump를 시도합니다. OOM과 재시작이 반복되는 서비스라면 프로세스마다 dump가 하나씩 생길 수 있다는 뜻이고, 이 누적 관리는 [5. 운영 적용 체크리스트](./5-production-checklist.md)에서 다룹니다.

## 언제 유용할까요?

공통점은 "재현할 수 없다"입니다.

- 며칠에 한 번씩 간헐적으로 죽는 서비스. 부하 테스트로는 재현되지 않는 누수
- 새벽에 죽어 있는 배치. 로그에는 OutOfMemoryError 한 줄만 남은 경우
- 상한 없는 캐시, 해제하지 않는 리스너처럼 코드 리뷰로 놓친 누적 버그
- 릴리스 이후 heap 사용량이 서서히 올라가는데 원인 코드를 못 찾는 경우

OOM은 예고 없이 오고, 증거는 재시작과 함께 사라집니다. 그래서 JVM이 죽는 순간에 스스로 증거를 남기게 만드는 것이 이 옵션의 존재 이유입니다. [3. OOM 순간의 dump 재현하기](./3-reproduce-oom-dump.md)에서 이 동작을 직접 재현합니다.
