# MariaDB 커넥션 실패 시 소켓 누수와 FD 고갈

## 개요

데이터베이스와 통신하는 애플리케이션에서 예외가 발생했을 때 **명시적으로 `Connection.close()`를 호출하지 않으면** 발생하는 TCP 소켓 누수와 FD(File Descriptor) 고갈 문제에 대해 설명합니다.

## 소켓 누수(Socket Leak)의 발생 원리

정상적인 환경에서는 애플리케이션이 데이터베이스 쿼리를 마치고 `Connection.close()`를 호출하면, 내부적으로 소켓이 닫히며 MariaDB 서버로 `FIN` 패킷이 전송됩니다.

하지만 예외 상황에서는 흐름이 꼬일 수 있습니다.

```txt
1. App ──TCP 연결──▶ MariaDB (소켓 생성, FD 할당)
2. App ──SELECT SLEEP(30)──▶ MariaDB
3. ... (3초 후 타임아웃 발생) ...
4. App: SocketTimeoutException 에러!
5. App: 예외 처리 중 실수로 conn.close() 누락!
6. MariaDB: 쿼리 계속 실행 중 (좀비 커넥션)
```

만약 타임아웃 같은 예외가 발생했을 때 개발자가 `catch` 블록이나 `finally`에서 `close()`를 확실하게 챙겨주지 않는다면, 해당 커넥션에 연결된 물리 소켓은 **영원히 닫히지 않고 `ESTABLISHED` 상태로 방치**됩니다.

> **💡 드라이버 특성 참고**
> 최신 버전의 드라이버(mariadb-java-client 2.7.4 이상)는 네트워크 예외가 발생하면 내부적으로 알아서 소켓을 닫아주어 애플리케이션의 실수를 어느 정도 방어해 줍니다. 하지만 2.7.2 이하 버전에서는 드라이버가 스스로 소켓을 닫지 않기 때문에, 애플리케이션 측에서 `close()`를 단 한 번이라도 누락하면 즉각 누수로 이어집니다.

## 소켓 누수의 파급 효과

### 1. 파일 디스크립터(FD) 고갈

리눅스 시스템은 파일과 네트워크 소켓을 모두 FD(파일 디스크립터)라는 정수 번호로 관리합니다.
누수된 소켓이 닫히지 않고 계속 쌓이게 되면, OS가 프로세스에 허용한 최대 FD 개수(`ulimit -n`)를 초과하게 됩니다. 결국 애플리케이션은 **`Too many open files`** 에러를 뱉으며 새로운 요청(HTTP 요청 포함)을 일절 받지 못하게 됩니다.

### 2. 서버 리소스 낭비 (CPU 상승)

누수된 소켓이 많아질수록 커널 레벨에서 활성 소켓을 감시하는 오버헤드가 커집니다. 애플리케이션 서버의 CPU 사용률이 점진적으로 상승하는 원인이 됩니다.

### 3. 데이터베이스 좀비 커넥션

애플리케이션은 이미 에러를 내고 끝났지만, 연결이 끊기지 않았으므로 DB 서버는 해당 쿼리를 끝까지 실행합니다. 이는 DB 서버의 `max_connections` 슬롯을 불필요하게 차지하고 락(Lock) 경합을 유발할 수 있습니다.

## 방어 대책 (해결 방법)

### 명시적인 close() 호출 보장

예외 발생 여부와 관계없이 자원을 무조건 해제해야 합니다. 자바에서는 `try-with-resources` 구문을 사용하면 코드가 블록을 벗어날 때 자동으로 `close()`를 호출해 주므로 누수를 완벽하게 방지할 수 있습니다.

```java
// 안전한 코드 작성법 (예외가 발생해도 무조건 close 됨)
try (Connection conn = DriverManager.getConnection(...);
     Statement stmt = conn.createStatement()) {
    stmt.executeQuery("SELECT SLEEP(30)");
} catch (SQLException e) {
    // 예외 처리
}
```
