# Docker Compose로 어떻게 재현할까

SSE는 브라우저나 클라이언트가 서버에 오래 연결되어 있어야 동작을 이해하기 쉽습니다. 로컬 Python만으로도 실행할 수 있지만, Docker Compose로 서버와 클라이언트를 나누면 연결 흐름이 더 선명해집니다. 왜 굳이 Compose로 나눠서 확인할까요?

## 서버와 클라이언트를 왜 분리할까

서버와 클라이언트를 같은 프로세스에서 실행하면 스트리밍 자체보다 실행 환경에 신경을 더 쓰게 됩니다. Compose에서는 `sse-server`가 HTTP 서버 역할을 하고, `sse-client`가 다른 컨테이너에서 이벤트를 읽습니다.

장점은 실제 네트워크 호출처럼 서비스 이름으로 접근할 수 있다는 점입니다. `sse-client`는 `http://sse-server:8000/events`로 서버를 찾습니다. 단점은 Docker가 필요하고, 단순 코드 실행보다 이미지 빌드 시간이 든다는 점입니다.

## 어떻게 실행할까

서버 컨테이너를 빌드하고 백그라운드로 실행합니다.

```bash
make up
```

서버 로그는 다음 명령으로 확인합니다.

```bash
make logs
```

실습을 끝낼 때는 컨테이너와 네트워크를 정리합니다.

```bash
make down
```

## 서버가 떠 있는지 어떻게 확인할까

브라우저에서 `http://localhost:8000`을 열면 간단한 EventSource 페이지가 뜹니다. 페이지는 `/events?limit=5&interval=1`에 연결하고 이벤트를 목록으로 추가합니다.

브라우저 없이 확인하려면 `curl`을 사용할 수 있습니다.

```bash
make curl
```

출력은 이벤트마다 `id`, `event`, `data`가 반복되는 형태입니다. 이벤트가 1초 간격으로 나오면 서버가 응답을 한 번에 끝내지 않고 스트림으로 흘려보내고 있다는 뜻입니다.

정리하면, Compose는 SSE의 핵심인 "열려 있는 HTTP 연결"을 서버 컨테이너와 클라이언트 컨테이너 사이에서 관찰하게 해 줍니다.
