# 클라이언트는 이벤트 스트림을 어떻게 확인할까

서버가 `text/event-stream`을 내려준다고 해서 클라이언트가 자동으로 의미 있는 이벤트를 얻는 것은 아닙니다. 줄 단위 스트림에서 어디까지가 이벤트 하나인지 읽어야 합니다. 그럼 클라이언트는 무엇을 확인해야 할까요?

## Python 클라이언트는 무엇을 출력할까

`sse-client` 컨테이너는 `httpx.stream()`으로 SSE 응답을 열고 줄 단위로 출력합니다.

```bash
make client
```

예상 출력은 다음과 비슷합니다.

```text
id: 1
event: tick
data: {"event_id": 1, "message": "server event stream is alive"}
id: 2
event: tick
data: {"event_id": 2, "message": "server event stream is alive"}
```

이 출력에서 중요한 것은 한 번의 JSON 응답이 아니라 여러 이벤트가 시간 차를 두고 내려온다는 점입니다. `limit=5`로 실행하면 다섯 번의 이벤트를 받은 뒤 연결이 닫힙니다.

## curl과 Python 클라이언트는 무엇이 다를까

`curl -N`은 버퍼링을 줄이고 서버가 보내는 줄을 바로 출력합니다. 그래서 SSE 원문 형식을 보기 좋습니다. Python 클라이언트는 애플리케이션 코드가 스트림을 어떻게 읽을 수 있는지 보여줍니다.

curl의 장점은 단순하다는 점입니다. 별도 클라이언트 코드를 믿지 않아도 서버 응답을 그대로 볼 수 있습니다. 단점은 재시도, 이벤트 파싱, 애플리케이션 처리 흐름을 표현하기 어렵다는 점입니다. Python 클라이언트의 장점은 실제 애플리케이션 코드로 확장하기 쉽다는 점이고, 단점은 SSE 전용 파서가 아니기 때문에 이번 실습에서는 원문 줄 확인까지만 다룬다는 점입니다.

## 어디까지 확인하면 충분할까

이 핸즈온의 완료 기준은 세 가지입니다.

1. Docker Compose로 FastAPI 서버가 실행된다.
2. `/events` 응답이 `text/event-stream`으로 열린다.
3. 클라이언트가 여러 이벤트를 순서대로 출력한다.

정리하면, 클라이언트 확인의 핵심은 "응답을 받았다"가 아니라 "같은 연결에서 이벤트가 순서대로 계속 내려왔다"입니다. 이 차이를 봐야 SSE를 일반 JSON API와 구분할 수 있습니다.
