# 운영 시나리오와 무중단 기준

TL;DR: TLS만 켜진 ElastiCache Valkey에 인증을 추가한다. 무인증 → AUTH token → RBAC → IAM 전용으로 한 단계씩 강화하되, 현재 트래픽을 받는 컨테이너의 Redis 오류가 한 건이라도 나오면 다음 단계를 멈춘다.

## 현재 상태와 목적

ElastiCache Valkey의 전송 중 암호화는 `required`이지만 인증은 꺼져 있어 모든 앱이 사용자 이름·password 없이 연결한다.

ElastiCache 설정부터 바꾸면 기존 앱이 `NOAUTH`나 `WRONGPASS`를 낸다. 그래서 인프라 변경과 앱 배포 사이에 구버전과 새 버전이 함께 붙을 수 있는 호환 구간을 두고 단계별로 넘어간다. 각 인증 방식의 개념은 [2-concepts.md](./2-concepts.md)에서 다룬다.

## 무중단의 기준

무중단은 현재 트래픽을 담당하는 컨테이너의 Redis 요청 오류가 0건인 상태다. 컨테이너를 새 버전으로 교체하는 배포는 허용하되, 배포 전후에 처리한 cache 요청에 오류가 없어야 한다.

Hurl은 cache API에 0.2초마다 `PUT`과 `GET`을 보낸다. HTTP 오류, Redis 연결 오류, 잘못된 응답이 나오면 gate가 끝난다. Terraform 변경 중 gate가 스스로 끝나면 다음 단계로 가지 않는다.

전환 완료 후 제거한 인증 방식을 확인하는 검증 컨테이너는 트래픽을 받지 않는다. 이 컨테이너에서 예상한 인증 오류가 나는 것은 무중단 실패로 계산하지 않는다.

## pod과 EC2 컨테이너의 대응

운영에서는 여러 pod이 rolling deployment로 교체된다. 이 실습은 pod을 EC2 한 대 안의 Docker 컨테이너로 대체한다. 컨테이너마다 host 포트를 달리해 구버전과 새 버전을 동시에 띄운다.

| host 포트 | 이미지 tag | 운영에서의 역할 | 실행 시점 |
| --- | --- | --- | --- |
| 8080 | `noauth-*` | 인증 정보가 없는 기존 앱 | 준비 단계 |
| 8081 | `auth-*` | AUTH token을 쓰는 새 앱 | AUTH 호환 구간부터 |
| 8082 | `iam-*` | IAM 인증을 쓰는 새 앱 | RBAC 호환 구간부터 |
| 8083 | `noauth-*` | 무인증이 막혔는지 보는 검증 앱 | AUTH 강제 후 |
| 8084 | `auth-*` | AUTH token이 막혔는지 보는 검증 앱 | IAM 전용 전환 후 |

트래픽 역할을 옮길 때는 새 컨테이너의 Hurl gate를 먼저 시작한다. 새 gate가 정상인 상태에서 기존 gate를 `Ctrl-C`로 끝내고 구버전 컨테이너를 종료한다. 이 순서가 운영의 rolling deployment(새 인스턴스를 readiness 통과 → 트래픽 전환 → 구버전 종료)에 대응한다.

## 다음 단계로 넘어가는 조건

새 컨테이너를 배포하고 cache 읽기·쓰기를 검증한다. 다음을 모두 만족할 때만 인증을 다음 단계로 강화한다.

- 새 컨테이너의 `cache-read.hurl`이 성공한다.
- 새 컨테이너의 Hurl gate가 기존 컨테이너 종료 전에 시작됐다.
- 인증 방식이 다른 구버전 컨테이너가 0대다.
- 현재 트래픽 컨테이너의 Hurl 오류가 0건이다.

하나라도 충족하지 못하면 다음 Terraform 변경을 보류하고, 실행 중인 컨테이너와 현재 인증 상태를 그대로 둔 채 로그와 ElastiCache 이벤트를 확인한다.
